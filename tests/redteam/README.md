# RT-1 Red Team Harness

Methodology: [`docs/RED_TEAM_METHODOLOGY.md`](../../docs/RED_TEAM_METHODOLOGY.md).
Decision criteria: [`docs/CATEGORIES_DECISION_CRITERIA.md`](../../docs/CATEGORIES_DECISION_CRITERIA.md).

## Running

```bash
# Full corpus, Layer 1 deterministic only (no LLM required)
POP_REDTEAM=1 POP_LLM_API_KEY= npx vitest --run tests/redteam

# Full corpus, all 5 paths (requires an OpenAI-compatible endpoint)
export POP_LLM_PROVIDER=openai
export POP_LLM_API_KEY=sk-...
export POP_LLM_MODEL=gpt-4o-mini-2024-07-18
POP_REDTEAM=1 npx tsx tests/redteam/run-corpus.ts --n=5 --concurrency=20

# B-class only (critical S1.1 input)
POP_REDTEAM=1 npx tsx tests/redteam/run-corpus.ts --filter=B
```

Artifacts land in `tests/redteam/runs/<ts>.jsonl`.

**The harness does NOT read `~/.config/pop-pay/.env`.** The production Layer-2 engine reads `POP_LLM_*` on its own; the harness only observes what the caller puts into the harness process's env. Founder credentials are safe.

## Regenerating the corpus

`corpus/attacks.json` is generated by Gemini CLI against the schema in `corpus/schema.json`. See [`corpus/GENERATION.md`](./corpus/GENERATION.md) for the prompt template and per-category quotas. After generation:

```bash
# Validate + dedup
npx tsx tests/redteam/validate-corpus.ts tests/redteam/corpus/attacks.json
```

## Contributing a payload

PRs against `corpus/attacks.json` must include:

- Deterministic `id` following the `[A-K]-NNNN` convention
- A concrete `expected` verdict (`block` or `approve`)
- `rationale` field explaining *why* the expected verdict is correct
- If contributing a benign counterpart (expected=approve), state which attack payload it mirrors

## Gotchas

- **Never log the corpus with secrets interpolated.** The harness constructs intents in-memory only.
- **LLM variance is expected.** N=5 runs per payload; if you see a >10% flip rate on a category, file a bug — the defense is a coin-toss.
- **Do not pin a CI API key.** The `requires:llm` tag exists so CI without a provider skips cleanly.
